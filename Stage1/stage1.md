# Stage 1: 构建基础 Agent 循环核心原理手册

## 1. 使用 LLM API 完成普通对话
### 核心原理
LLM 的本质是**基于概率的文本补全机**。在对话模式下，它通过接收一个**消息序列（Message Sequence）**并预测下一个回复。

*   **无状态性 (Statelessness)**：API 本身不记录历史。每一次调用都是独立的。为了实现“对话”，必须将之前的历史记录全部重新发送给模型。
*   **角色定义 (Message Roles)**：
    *   **System Message**：设定全局基调、规则和约束（Agent 的“人格”）。
    *   **Human Message**：用户的即时输入。
    *   **AI Message**：模型之前的回复。
*   **Token 机制**：文本被切割为 Token。理解上下文窗口（Context Window）限制是对话设计的关键。

### 注意要点
*   **Temperature (温度)**：控制随机性。Agent 逻辑通常建议设为 `0`，以获得稳定的推理路径。
*   **上下文修剪**：当对话过长，需要通过总结（Summary）或丢弃旧消息来保护上下文空间。

## 2. 让模型输出结构化 JSON
### 核心原理
LLM 默认输出非结构化文本。实现结构化输出有四种主要底层方法：

1.  **Prompt Engineering (最原始)**：在提示词中写“请只返回 JSON”。原理完全依赖模型的指令遵循能力。
2.  **JSON Mode**：API 层面强制校验输出。模型生成时会确保符合 JSON 语法，但**不保证**符合你的特定 Schema（字段名可能乱写）。
3.  **Function Calling / Tool Calling (主流)**：模型被训练为“填充表单”。它输出的是函数参数，天然具备结构化特征。
4.  **Strict Mode / Constrained Decoding (最强)**：底层推理引擎利用**语法状态机（GBNF）**。在生成每个字符时，强行禁止输出不符合 Schema 定义的字符。

### 注意要点
*   **Pydantic 角色**：Pydantic 不仅用于验证，它还负责将 Python 类自动生成模型能理解的 JSON Schema 描述。
*   **幻觉字段**：即使是结构化输出，模型也可能生成 Schema 之外的字段，必须在解析层做过滤。

## 3. 工具函数定义 (Tool Definition)
### 核心原理
工具定义本质上是**元数据提取（Metadata Extraction）**。LLM 并不直接接触你的 Python 代码，它只看你的**接口说明书**。

*   **语义映射**：模型将用户的自然语言意图（“查天气”）与工具的 `description` 进行语义匹配。
*   **Schema 注入**：当你定义一个工具，它的函数名、参数名、参数类型、Docstring 都会被转化为 JSON Schema，塞进 System Prompt 或是 API 的 `tools` 字段中。

### 实现方式
*   **装饰器模式 (`@tool`)**：自动提取函数名和注释。
*   **类定义模式 (`BaseTool`)**：适合需要复杂状态或逻辑的工具。
*   **结构化工具 (`StructuredTool`)**：支持多参数复杂输入。

## 4. 解析 Tool Call / Function Call
### 核心原理
这是从“文本生成”到“意图动作”的跨越。

*   **停止信号 (Stop Reason)**：当模型决定调用工具时，生成的停止原因是 `tool_calls` 而非 `end_of_turn`。
*   **参数实例化**：模型输出的 `args` 是字符串格式。解析器必须将其 `json.loads` 化，并与工具定义的参数模型进行比对。
*   **并行调用原理**：现代模型支持一次返回多个 `tool_call`。解析层必须具备循环处理列表的能力，并为每个调用分配/记录唯一的 `id`。

### 注意要点
*   **参数注入风险**：模型生成的参数可能类型不符（比如该传 int 传了 string），解析层需要具备类型转换的鲁棒性。

## 5. 执行工具并喂回结果 (The ReAct Loop)
### 核心原理
这是 Agent 的“神经反射弧”。模型处于等待状态，直到它接收到工具的反馈。

*   **ToolMessage 角色**：这是一种特殊消息。它必须包含 `content`（工具执行结果）和 `tool_call_id`。
*   **对暗号机制**：LLM 收到 `ToolMessage` 后，会根据 `tool_call_id` 将结果与之前的请求匹配。
*   **循环迭代**：
    1.  LLM 生成 Call -> 暂停。
    2.  代码执行 Call -> 得到 Observation。
    3.  Observation 喂回 LLM -> 恢复推理。
    4.  LLM 判断：结果够了吗？不够继续 Call，够了输出最终结论。

### 注意要点
*   **序列化**：工具返回的复杂 Python 对象（如数据库连接、数据框）必须转化为字符串才能喂给模型。

## 6. Agent 循环的保护机制
### 核心原理

#### A. 最大步数 (Recursion Limit / Max Iterations)
*   **原理**：在图中设置计数器。每经过一次节点转换，计数器减一。
*   **必要性**：防止模型陷入“工具 A 需要工具 B，工具 B 又需要工具 A”的逻辑死循环，耗尽 Token 费用。

#### B. 超时处理 (Timeout)
*   **原理**：利用异步 IO (`asyncio.wait_for`) 或线程计时。
*   **注意点**：LLM 响应可能很快，但某些工具（如搜索、网页爬取）可能极慢。需要针对不同工具设置不同的超时阈值。

#### C. 错误处理与自我修复 (Self-Correction)
*   **重试逻辑**：针对网络波动的指数退避重试。
*   **错误反馈**：当工具报错，将 Python Traceback 的简化版本作为 `ToolMessage` 发回给模型。
    *   **原理**：模型看到报错信息（如“缺少参数 X”）后，有能力在下一轮迭代中修正参数重新调用。
